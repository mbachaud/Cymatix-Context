## 2. What the investigation found (this is NOT a clean version bump)

**BLOCKER — the numbers-producing code and the shipped release have DIVERGED into siblings.**
- PyPI public latest = **0.6.2**. Git: 0.6.2 = `origin/master` 3173c47, **never tagged** (newest tag
  v0.6.1); it was published via manual/workflow_dispatch.
- **Local master is STALE at 0.5.0** (826d053, behind origin).
- The **worktree** `perf/dense-prefilter-via-splade-candidates` (191ce31 + dirty) — which produced
  ALL our numbers — branched off the **stale 826d053**, NOT off 0.6.2. `merge-base(worktree, 0.6.2)
  = 826d053`. So **worktree and 0.6.2 are siblings; neither contains the other.**
- The worktree's uncommitted RAM levers (shared-codec singleton, fp16 matrix, PRAGMA caps, concurrent
  fan-out, tagger fix, FTS5-skip, IN-batching) are **parallel re-implementations** of fixes 0.6.2
  **already shipped** (#173 shared codec d5ade41, #175 HELIX_MEM_PROFILE, concurrent fan-out f30f372,
  #163 IN-batching). They will conflict on a merge/rebase onto 0.6.2.

**Config — the "fixed pipeline" is GATED OFF on a bare install (byte-identical to baseline).** To
reproduce our numbers you must set ALL of:
- `HELIX_QUESTION_DENSE=1` (env-only, no config key; default OFF → dense tier uses tag-bag)
- `HELIX_SEMANTIC_ARM=1` (env-only; gates BOTH semantic broaden routing AND the semantic dense weight;
  applies ONLY to `query_type=="semantic"`)
- dense weight **16** — NOT a default anywhere (dataclass=4.0; semantic-scoped=12.0); needs
  `HELIX_SEMANTIC_DENSE_WEIGHT=16` or a `[retrieval] semantic_dense_additive_weight=16` toml edit
- `HELIX_SHARD_WORKERS=8` (env-only; default serial=1; recall-IDENTICAL, latency knob only)
- the SPLADE-on / mg10 / rerank profile (`F:/tmp/helix_splade_on_mg10_rerank.toml`) — but note this
  toml as-written does NOT set the four knobs above; it must be augmented.

**Provenance gaps (must resolve before external sharing):**

## 3. Decisions required (sign-off gates)

**D1 — Release shape (drives everything):**
- **Shape 1 — frozen validation artifact (RECOMMENDED).** Tag `v0.6.3` on the worktree (after
  committing), build a wheel, hand Onyx {wheel + tag + pinned config + VALIDATION.md + numbers with
  provenance}. **NOT** published to public PyPI. Sidesteps the v0.6.2 reconciliation entirely (a
  validation snapshot need not carry 0.6.2's RAM work — that's mem/latency, irrelevant to recall/
  correctness). Fastest, lowest-risk, most honest "this is exactly what produced the numbers."
- **Shape 2 — public PyPI 0.6.3 (reconciled).** Sync to 0.6.2, cherry-pick the genuinely-new product
  changes onto it (question-dense + query_type thread + per-gene budget + rerank-on-/context fix +
  semantic arm), DROP the duplicate RAM re-impls (0.6.2 has them). Full test suite. Correct + durable
  but real git surgery + conflict resolution + irreversible publish.
- **Shape 3 — worktree as public 0.6.3 as-is. NOT RECOMMENDED** — regresses 0.6.2's shipped RAM/mem
  work (HELIX_MEM_PROFILE, canonical shared codec) for all upgrading users.

**D2 — Config posture (how Onyx reproduces):**
- **B1 — default-off + pinned profile (RECOMMENDED).** Ship baseline defaults; hand Onyx a `helix.toml`
  + env block that activates the fixed pipeline. Doesn't bake the refuted arm into the product default.
- **B2 — flip the fixed pipeline on by default.** Bakes in the smoke-refuted semantic arm + changes
  behavior for all users. Not recommended.

**D3 — Semantic arm in the validated config:**
- **Include it (RECOMMENDED, documented).** Matches the numbers exactly; the full-500 A/B showed it
  net-safe. Document honestly that it's semantic-scoped and small-positive.
- **Re-baseline without it** (question-dense only) — cleaner "no refuted lever" story, but requires a
  fresh ~10h capture + recall run to get new numbers.
